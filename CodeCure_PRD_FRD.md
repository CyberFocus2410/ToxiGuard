# **1. Executive Summary & Product Vision**
## **1.1 Product Overview**
CodeCure is a full-stack, AI-powered drug toxicity prediction web application. Given a molecule's SMILES (Simplified Molecular Input Line Entry System) string, the platform predicts toxicity probability, categorizes risk, identifies the top contributing molecular features via SHAP analysis, and provides a human-readable biological interpretation generated by Claude AI. The system stores all predictions in a structured database and exposes a real-time analytics dashboard.

## **1.2 Problem Statement**
Drug toxicity is responsible for approximately 30% of drug development failures in clinical trials, costing pharmaceutical companies millions of dollars and years of effort per failed compound. Traditional toxicity screening requires wet-lab experiments which are slow, expensive, and resource-intensive. An in-silico (computational) prediction tool can catch high-risk compounds early in the pipeline — before expensive trials begin.

## **1.3 Proposed Solution**
CodeCure addresses this gap by combining:

- A trained XGBoost machine learning model on the Tox21 dataset (8,000+ compounds)
- RDKit-based molecular descriptor computation (20+ features per compound)
- SHAP-based explainability for per-prediction feature attribution
- Claude AI for translating predictions into plain-language biological insights
- Antigravity as the backend orchestration and database layer
- A React-based frontend dashboard for interactive exploration

## **1.4 Business Goals & Success Metrics**

|**Metric**|**Target**|**Rationale**|
| :- | :- | :- |
|Model Accuracy|≥ 85%|Minimum threshold for scientific credibility|
|F1 Score|≥ 0.84|Balance precision and recall for toxic compounds|
|AUC-ROC|≥ 0.91|Discriminative ability across thresholds|
|API Response Time|< 2 seconds|User experience and demo readiness|
|Frontend Load Time|< 3 seconds|Performance benchmark|
|Database Write Success Rate|99\.9%|Data integrity requirement|

## **1.5 Scope**
### **In Scope**
- ML training pipeline (Tox21 dataset, XGBoost, SHAP)
- REST API backend on Antigravity (4 endpoints)
- PostgreSQL database with 3 tables
- React frontend: prediction form + analytics dashboard
- Claude API integration for biological interpretation
- Deployment on Vercel (frontend) and Antigravity (backend)
- Unit + integration test suites
### **Out of Scope**
- User authentication and multi-tenancy (deferred to v2)
- 3D molecular visualization
- Multi-target toxicity assay selection
- SMILES drawing tool / molecule editor


# **2. System Architecture**
## **2.1 High-Level Architecture**
CodeCure follows a three-tier architecture with a separate ML execution layer:

|**Tier**|**Technology**|**Responsibility**|
| :- | :- | :- |
|Frontend (Presentation)|React.js + Recharts + Axios|User input, result display, dashboard, visualizations|
|Backend (Application)|Antigravity Visual Builder|REST API routing, workflow orchestration, DB writes|
|ML Execution Layer|Python (XGBoost + RDKit + SHAP)|Descriptor computation, model inference, SHAP values|
|AI Interpretation Layer|Claude API (claude-sonnet)|Biological explanation generation|
|Database (Persistence)|PostgreSQL via Antigravity|Predictions, model metadata, feature importance|

## **2.2 Data Flow Diagram**
The end-to-end prediction flow follows this sequence:

1. User enters a SMILES string in the React frontend and clicks Predict.
1. Frontend POSTs { smiles, user\_id } to POST /api/predict on Antigravity.
1. Antigravity validates the input format (basic string checks).
1. Antigravity calls the uploaded Python ML function with the SMILES string.
1. Python computes 20+ RDKit descriptors, scales them, runs XGBoost inference, and computes SHAP values.
1. Python returns { toxicity\_probability, toxicity\_category, risk\_factors, confidence }.
1. Antigravity calls Claude API with prediction data to generate a biological interpretation (≤150 words).
1. Antigravity writes the full result to the predictions table in PostgreSQL.
1. Antigravity returns the enriched JSON response to the frontend.
1. React renders the toxicity score, risk factor table, bar chart, and AI interpretation.

# **4. Backend API Specification**
All four endpoints below must be implemented in Antigravity. The base URL is the Antigravity project URL (e.g., https://your-project.antigravity.dev). All endpoints return JSON. Errors follow { error: string, code: number } format.

## **4.1 POST /api/predict**
**The core prediction endpoint. This is the most complex workflow.**

|**Method**|POST|
| :- | :- |
|**Path**|/api/predict|
|**Auth**|None (v1) — API key header in future|
|**Content-Type**|application/json|

### **Request Body**

|**Field**|**Type**|**Required**|**Description**|
| :- | :- | :- | :- |
|smiles|string|Yes|SMILES string of the molecule (e.g., 'CCO' for ethanol)|
|user\_id|string|No|Caller identifier; stored for history lookup|

### **Antigravity Workflow Steps (in order)**
1. Receive and parse JSON body.
1. Validate smiles field is non-empty string. Return HTTP 400 if missing.
1. Call Python custom function predict\_toxicity\_api(smiles). Return HTTP 500 on runtime error.
1. Call Claude API with prediction result to generate interpretation text (max 200 tokens).
1. INSERT all fields into predictions table. Record id is captured.
1. Return HTTP 200 with assembled JSON response.

### **Success Response (HTTP 200)**

|**Field**|**Type**|**Description**|
| :- | :- | :- |
|id|UUID|Database record ID of the stored prediction|
|toxicity\_probability|float|Value between 0.0 and 1.0; higher = more toxic|
|toxicity\_category|string|One of: 'High' (≥0.7), 'Medium' (0.4–0.69), 'Low' (<0.4)|
|risk\_factors|array|Top 5 objects: { feature, value, importance, contribution }|
|confidence|float|Max softmax probability from the model|
|interpretation|string|Claude-generated plain-English biological explanation|

### **Error Responses**

|**HTTP Code**|**Trigger Condition**|**Response Body**|
| :- | :- | :- |
|400|Missing or empty smiles field|{ error: 'Invalid SMILES format', code: 400 }|
|500|Python ML pipeline exception|{ error: 'ML pipeline error', code: 500 }|
|503|Claude API unavailable|{ error: 'Interpretation unavailable', code: 503 }|

## **4.2 GET /api/predictions/:user\_id**

|**Method**|GET|
| :- | :- |
|**Path**|/api/predictions/:user\_id|
|**Path Param**|user\_id — the user whose history to retrieve|

### **Antigravity Workflow Steps**
1. Extract user\_id from path.
1. Query: SELECT \* FROM predictions WHERE user\_id = :user\_id ORDER BY created\_at DESC LIMIT 50.
1. Return array of prediction records (empty array if none found).

### **Success Response (HTTP 200)**
Returns a JSON array of prediction objects. Each object contains all columns from the predictions table. Empty array [] if no records exist for the given user\_id.

|**Field**|**Type**|**Example**|

|**Method**|GET|
| :- | :- |
|**Path**|/api/feature-importance|

# **5. ML Pipeline Functional Requirements**
## **5.1 Python Environment Requirements**
The following packages must be installable in the Antigravity Python runtime or local execution environment:

|**Package**|**Version**|**Purpose**|
| :- | :- | :- |
|scikit-learn|≥ 1.3|Preprocessing, StandardScaler, train/test split|
|xgboost|≥ 1.7|Gradient boosted classifier|
|shap|≥ 0.43|SHAP value computation (TreeExplainer)|
|rdkit|≥ 2023.03|Molecular descriptor computation from SMILES|
|pandas|≥ 2.0|DataFrame operations for dataset handling|
|numpy|≥ 1.24|Array operations|
|pickle|stdlib|Model serialization/deserialization|

## **5.2 Dataset Requirements**
- Dataset: Tox21 (available on Kaggle as tox21\_data.csv)
- Target column: Toxicity (binary: 0 = non-toxic, 1 = toxic)
- Input column: SMILES (chemical structure string)
- Expected rows: ~8,000 compounds after deduplication and null removal

## **5.3 Molecular Descriptors (20 Required)**
The following RDKit descriptors must be computed for each compound. All are available via rdkit.Chem.Descriptors:

|**Descriptor**|**RDKit Function**|**Biological Relevance**|
| :- | :- | :- |
|LogP|Descriptors.MolLogP|Lipophilicity; membrane permeability predictor|
|MolWt|Descriptors.MolWt|Molecular weight; absorption/distribution|
|NumRotatableBonds|Descriptors.NumRotatableBonds|Flexibility; Lipinski compliance|
|NumHDonors|Descriptors.NumHDonors|H-bond donors; bioavailability|
|NumHAcceptors|Descriptors.NumHAcceptors|H-bond acceptors; solubility|
|NumRings|Descriptors.RingCount|Ring count; metabolic stability|
|NumAromaticRings|Descriptors.NumAromaticRings|Aromaticity; mutagenicity risk|
|TPSA|Descriptors.TPSA|Topological polar surface area; CNS penetration|
|EState\_VSA1–6|Descriptors.EState\_VSA1-6|Electronic state surface area bins|
|LabuteASA|Descriptors.LabuteASA|Labute's approximate surface area|
|PEOE\_VSA1–5|Descriptors.PEOE\_VSA1-5|Partial charge surface area bins|

## **5.4 Training Pipeline Requirements**
### **FR-ML-01: Data Preprocessing**
- Load CSV file from path: tox21\_data.csv
- Drop rows with null SMILES or null Toxicity values
- Remove duplicate SMILES entries
- Validate SMILES using RDKit Chem.MolFromSmiles(); discard invalid entries
### **FR-ML-02: Feature Engineering**
- Compute all 20 descriptors for each valid molecule
- Store as a feature matrix X; store target as y
- Apply StandardScaler to X; fit only on training set
### **FR-ML-03: Train/Test Split**
- Split: 80% training, 20% test (random\_state=42 for reproducibility)
- Apply 5-fold cross-validation on training set only
### **FR-ML-04: Model Training**
- Classifier: XGBClassifier with parameters: n\_estimators=300, max\_depth=6, learning\_rate=0.1, use\_label\_encoder=False, eval\_metric='logloss'
- Train on full training split after cross-validation
### **FR-ML-05: Evaluation**
- Compute accuracy, F1 score (weighted), AUC-ROC on held-out test set
- Print cross-validation mean and std to stdout
- Insert evaluation results into model\_metadata table via API after training
### **FR-ML-06: SHAP Analysis**
- Initialize shap.TreeExplainer(model) after training
- Compute mean |SHAP| per feature across the training set for global importance
- Insert top 10 features into feature\_importance table via API after training
### **FR-ML-07: Model Serialization**
- Save model, scaler, and shap\_explainer together as a single pickle file: toxicity\_model.pkl
- Upload this file to Antigravity as a custom function resource before deploying the prediction API

## **5.5 Prediction Function Specification**
The function predict\_toxicity\_api(smiles: str) must be the callable entry point for Antigravity. It must:

1. Load the model from the mounted file path (cache after first load to avoid re-loading on each call).
1. Call compute\_descriptors(smiles) using RDKit.
1. Raise ValueError('Invalid SMILES') if RDKit cannot parse the string.
1. Scale features with the loaded StandardScaler.
1. Call model.predict\_proba(features\_scaled) and extract the positive class probability.
1. Run shap\_explainer.shap\_values(features\_scaled) and compute per-feature attribution.
1. Sort features by absolute SHAP value; return top 5.
1. Return a dict with keys: toxicity\_probability, toxicity\_category, risk\_factors, confidence.

## **11.2 Hackathon Judge Questions & Answers**

|**Question**|**Recommended Answer**|
| :- | :- |
|Why XGBoost over deep learning?|XGBoost trains in seconds on tabular descriptors, is fully SHAP-compatible, and achieves state-of-the-art performance on structured cheminformatics data. Neural nets require more data and lose interpretability.|
|How did you validate the model?|5-fold cross-validation on the training set, plus a held-out 20% test set evaluated on accuracy, F1, and AUC-ROC. Results are stored in the database for transparency.|
|What is the false positive rate?|Computable from the confusion matrix on the test set. We optimize for recall (catching toxic compounds) at the cost of some precision.|
|Can it handle unseen scaffolds?|XGBoost generalizes to novel compounds via the descriptor space, but predictions for chemotypes far from the training distribution have lower confidence scores.|
|Production deployment?|Antigravity scales the API horizontally. The model can be retrained periodically via the same pipeline and re-uploaded as a new version with metadata tracked in the database.|
